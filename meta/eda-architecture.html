Key Design Decisions
DecisionRationale
racky-stylist owns a garment read model Consumes garment.tagged / garment.deleted to maintain local copies — outfit generation needs no runtime HTTP call to racky-closet, keeping services fully decoupled
Gateway never touches Kafka Gateway is a thin HTTP router; event production is the responsibility of the domain service that owns the state change
user.deleted fans out Cascade deletes (garments, outfits) are handled by each service consuming the event independently — no service orchestrates another's cleanup
Auth is synchronous Login/register are inherently request-response — forcing them async adds latency and complexity with no benefit; EDA doesn't mean async everywhere
POST /garments returns 202 Tagging is async; frontend polls GET /garments/:id and reads the status field (pending → tagged) — eliminates the UX gap in the original design
racky-tagger has a DLQ If Claude returns unparseable JSON or the API is down, the event goes to a dead letter queue for retry/alerting rather than silently dropping
Consumers must be idempotent Kafka at-least-once delivery means garment.tagged may arrive twice — upsert logic in racky-closet and racky-stylist prevents duplicate state
